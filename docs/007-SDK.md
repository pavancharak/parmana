\# Purpose



This document defines the Software Development Kit (SDK) architecture for the Parmana platform.



The SDK provides the primary developer interface to Parmana.



It exposes business capabilities while hiding implementation complexity.



\---



\# SDK Philosophy



The SDK is execution-centric.



Developers interact with ExecutionTransactions.



Developers never interact directly with:



\* Receipts

\* Attestations

\* Hashes

\* Signatures

\* Crypto Providers

\* Verification Algorithms



Those remain internal implementation details.



\---



\# Design Principles



1\. Execution-first

2\. Domain-oriented

3\. Type-safe

4\. Deterministic

5\. Idempotent where applicable

6\. Cryptographically transparent

7\. Forward compatible



\---



\# Primary Object



The SDK revolves around one object.



```text

Parmana

```



Applications instantiate one client.



Example



```typescript

const parmana = new Parmana(config);

```



\---



\# Primary Workflow



The SDK models the complete execution lifecycle.



```text

Create Transaction

&#x20;       │

&#x20;       ▼

Authorize

&#x20;       │

&#x20;       ▼

Execute

&#x20;       │

&#x20;       ▼

Verify

&#x20;       │

&#x20;       ▼

```

\# Core Operations



The SDK exposes a minimal set of operations.



```typescript

createTransaction()



authorize()



execute()



verify()



getTransaction()



getEvidence()



getTrustReport()

```



These operations are stable.



\---



\# Transaction Lifecycle



Example



```typescript

const transaction = await parmana.createTransaction({

&#x20; authority,

&#x20; intent

});



await parmana.authorize(transaction.id);



await parmana.execute(transaction.id);



const verification = await parmana.verify(transaction.id);



const report = await parmana.getTrustReport(transaction.id);

```



This represents the complete execution lifecycle.



\---



\# Transaction Object



The SDK exposes a simplified transaction model.



```typescript

interface ExecutionTransaction {



&#x20;   id: string;



&#x20;   authority: Authority;



&#x20;   intent: Intent;



&#x20;   status: TransactionStatus;



}

```



Implementation artifacts remain internal.



\---



\# Verification Result



Verification returns a stable structure.



```typescript

interface VerificationResult {



&#x20;   status:

&#x20;       | "PASS"

&#x20;       | "FAIL"

&#x20;       | "INCOMPLETE"

&#x20;       | "UNKNOWN";



&#x20;   verifiedAt: Date;



&#x20;   reportId: string;



}

```



The structure may grow.



Existing fields remain stable.



\---



\# Trust Report



Trust Reports provide the primary human-readable result.



```typescript

const report =

&#x20;   await parmana.getTrustReport(transaction.id);

```



Example



```text

Execution Trust Report



Authority       PASS

Intent          PASS

Authorization   PASS

Execution       PASS

Evidence        PASS

Integrity       PASS



Overall         PASS

```

Example



```typescript

try {



&#x20;   await parmana.verify(id);



}



catch(error) {



&#x20;   if(error.code==="VERIFICATION\_FAILED") {



&#x20;   }



}

```



SDKs never expose raw infrastructure errors.

\# Advanced Access



Advanced APIs may expose implementation artifacts.



Examples




\---



\# Dependency Rules



Applications depend only on:



```text

Application

&#x20;       │

&#x20;       ▼

Parmana SDK

&#x20;       │

&#x20;       ▼

HTTP API

&#x20;       │

&#x20;       ▼

Parmana Platform

```



Applications never depend on internal platform modules.



\---



\# Versioning



SDK versions are independent of API versions.



Breaking SDK changes require a major version increment.



The SDK maintains compatibility with supported API versions.



\---



\# Extension Model



SDKs support extension through well-defined interfaces.



Examples include:



\* Authentication providers

\* Retry strategies

\* Logging

\* Telemetry

\* Event handlers



Business semantics remain unchanged.



\---



\# Architectural Constraints



The SDK:



\* exposes business concepts

\* hides implementation artifacts

\* remains independent of cryptographic algorithms

\* remains independent of storage implementation

\* remains stable across runtime evolution



\---



\# Success Criterion



The SDK succeeds when developers can integrate trusted execution into their applications using a small, stable, and intuitive API without understanding the platform's internal cryptographic, evidence, or verification mechanisms.
