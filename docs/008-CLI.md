\# Purpose



This document defines the Command Line Interface (CLI) architecture of the Parmana platform.



The CLI is the primary operational interface for developers, operators, auditors, and security teams.



The CLI demonstrates the Execution Trust model.



It is not a debugging tool.



\---



\# CLI Philosophy



The CLI exposes business capabilities.



The CLI hides implementation complexity.



Every command should answer a trust question rather than expose internal platform mechanics.



\---



\# Design Principles



1\. Execution-first

2\. Human-readable

3\. Machine-readable

4\. Deterministic

5\. Scriptable

6\. Stable

7\. Versioned



\---



\# Primary Command



The root command is:



```text id="m7jqax"

parmana

```



\---



\# Command Groups



The CLI exposes the following command groups.



```text id="8zrjgx"

parmana transaction



parmana verify



parmana evidence



parmana report



parmana replay



parmana config



parmana version

```



Each command group represents a business capability.



\---



\# Transaction Commands



Create a transaction.



```text id="kydwsm"

parmana transaction create

```



Retrieve a transaction.



```text id="m3d1jy"

parmana transaction get <transactionId>

```



List transactions.



```text id="s99rgh"

parmana transaction list

```



\---



\# Verification Commands



Verify an execution.



```text id="4lfkqb"

parmana verify <transactionId>

```



Verification is deterministic.



Running the same command against unchanged evidence produces the same result.



\---



\# Evidence Commands



Inspect evidence.



```text id="8gy7q5"

parmana evidence show <transactionId>

```



Export evidence.



```text id="gyhm6g"

parmana evidence export <transactionId>

```



Evidence is read-only.



\---

\# Replay



Replay verification.



```text id="qu7w8n"

parmana replay <transactionId>

```



Replay produces the same verification result using recorded evidence.

\# Exit Codes



The CLI uses deterministic exit codes.



| Code | Meaning               |

| ---: | --------------------- |

|    0 | Success               |

|    1 | Verification Failed   |

|    2 | Invalid Input         |

|    3 | Transaction Not Found |

|    4 | Evidence Missing      |

|    5 | Internal Error        |



Exit codes remain stable.

```text id="v4dn17"

debug
```

Debug mode may expose implementation artifacts.



Normal mode does not.

\---



\# Architectural Constraints



The CLI:



\* exposes business concepts

\* hides implementation details

\* remains deterministic

\* remains scriptable

\* remains stable across runtime evolution



The CLI never becomes a replacement for internal debugging tools.



\---



\# Success Criterion



The CLI succeeds when a developer, operator, or auditor can create, inspect, verify, replay, and report on trusted executions using a small, consistent set of commands without understanding the platform's internal implementation details.
