\# Purpose



This document defines the public API architecture of the Parmana platform.



The API exposes business capabilities.



The API does not expose internal implementation artifacts.



Public APIs remain stable even when internal implementations evolve.



\---



\# API Philosophy



The Parmana API is resource-oriented.



Resources represent business concepts.



Implementation artifacts remain internal.



\---



\# API Principles



1\. APIs expose domain concepts.

2\. APIs are deterministic.

3\. APIs are versioned.

4\. APIs are idempotent where applicable.

5\. APIs return immutable resources.

6\. APIs never expose cryptographic implementation details unless explicitly requested.

7\. APIs are stable across implementation changes.



\---



\# Resource Model



The API exposes five primary resources.



```text

ExecutionTransaction



Execution



Evidence



Verification



TrustReport

```



Everything else is internal.



\---



\# Resource Relationships



```text

ExecutionTransaction

&#x20;       │

&#x20;       ├── Execution

&#x20;       ├── Evidence

&#x20;       ├── Verification

&#x20;       └── TrustReport

```



The API revolves around ExecutionTransaction.



\---



\# API Versioning



Every endpoint is versioned.



Example



```text

/v1/

```



Future versions:



```text

/v2/

/v3/

```



Breaking changes require a new version.



\---



\# Primary Resources



\## ExecutionTransaction



Represents the complete execution lifecycle.



Endpoints



```text

POST   /v1/transactions



GET    /v1/transactions/{transactionId}



GET    /v1/transactions



DELETE /v1/transactions/{transactionId}

```



Delete behavior is implementation-defined.



Production systems may archive rather than delete.



\---



\## Execution



Represents recorded execution.



Endpoints



```text

GET /v1/transactions/{transactionId}/execution

```



Execution is read-only after creation.



\---



\## Evidence



Represents immutable evidence.



Endpoints



```text

GET /v1/transactions/{transactionId}/evidence



GET /v1/evidence/{evidenceId}

```



Evidence is append-only.



Evidence is never modified.



\---



\## Verification



Represents verification results.



Endpoints



```text

POST /v1/transactions/{transactionId}/verify



GET  /v1/transactions/{transactionId}/verification

```



Verification is deterministic.



Repeated verification over unchanged evidence produces identical results.



\---



\## Trust Report



Represents the human-readable verification outcome.



Endpoints



```text

GET /v1/transactions/{transactionId}/trust-report

```



Trust Reports are immutable snapshots.



\---



\# Transaction Creation



Example



```http

POST /v1/transactions

```



Response



```json

{

&#x20; "transactionId": "txn\_01J...",

&#x20; "status": "CREATED",

&#x20; "createdAt": "2026-06-25T12:00:00Z"

}

```



The response establishes the ExecutionTransaction.



\---



\# Verification Flow



```text

POST /verify

&#x20;       │

&#x20;       ▼

Load ExecutionTransaction

&#x20;       │

&#x20;       ▼

Evaluate Invariants

&#x20;       │

&#x20;       ▼

Generate Verification Report

&#x20;       │

&#x20;       ▼

Return Verification Result

```



Verification never modifies execution.



\---



\# Verification Result



Example



```json

{

&#x20; "status": "PASS",

&#x20; "verifiedAt": "2026-06-25T12:10:00Z",

&#x20; "invariants": {

&#x20;   "authority": "PASS",

&#x20;   "intent": "PASS",

&#x20;   "authorization": "PASS",

&#x20;   "execution": "PASS",

&#x20;   "evidence": "PASS",

&#x20;   "integrity": "PASS"

&#x20; }

}

```



The structure remains stable.



Additional invariant details may be added without breaking compatibility.



\---



\# Error Model



Errors use a consistent format.



Example



```json

{

&#x20; "error": {

&#x20;   "code": "VERIFICATION\_FAILED",

&#x20;   "message": "Execution does not match authorized intent."

&#x20; }

}

```



Every error contains:



\* code

\* message

\* correlationId



Optional:



\* details

\* documentationReference



\---



\# Idempotency



The following operations must be idempotent:



\* Verification

\* Trust Report retrieval

\* Resource retrieval



Transaction creation may support an Idempotency-Key header.



\---



\# Filtering



Collection endpoints support filtering.



Examples



```text

GET /v1/transactions?status=VERIFIED



GET /v1/transactions?authority=user123



GET /v1/transactions?intent=payment



GET /v1/transactions?createdAfter=2026-01-01

```



Filtering behavior is implementation-defined.



\---



\# Pagination



Collection endpoints use cursor-based pagination.



Example



```text

GET /v1/transactions?cursor=...

```



The API avoids offset-based pagination for large datasets.



\---



\# API Stability



The following resources are permanent:



\* ExecutionTransaction

\* Execution

\* Evidence

\* Verification

\* TrustReport



New fields may be added.



Existing fields must not change meaning.



\---



\# Internal Resources



The following are implementation artifacts and are not exposed as primary resources:



\* Decision

\* Policy Snapshot

\* Attestation

\* Receipt

\* Signature

\* Hash

\* Ledger Record



They may appear as embedded evidence when explicitly requested.



\---



\# Security



The API supports:



\* Authentication

\* Authorization

\* Audit Logging

\* Request Signing

\* Rate Limiting



Security mechanisms are implementation-specific.



\---



\# Architectural Constraints



The API:



\* exposes business concepts

\* hides implementation details

\* remains independent of cryptographic algorithms

\* remains independent of storage implementations

\* remains compatible with future execution engines



\---



\# Success Criterion



The API succeeds when clients can create, inspect, verify, and audit trusted executions without requiring knowledge of the platform's internal implementation artifacts or cryptographic mechanisms.
